## Summary

This phase is **post-cutover SEO growth** for the Laravel Blade vitrine. The codebase audit confirms the technical baseline from Phase 1 is solid (`<x-seo.meta>` resolves canonical/og via `url()->current()`, `twitter:card` present, sitemap dynamic via `spatie/laravel-sitemap`, `Schema::article()` already on blog). The work splits into four clean buckets: **(1) schema enrichment** (multi-type + founder + email), **(2) content depth** (expand 2 thin pages + author 4 service pages + 4 city hubs), **(3) static-asset image optimization** (12 .jpg → WebP/AVIF), and **(4) discoverability plumbing** (`llms.txt`, `BreadcrumbList`, blog `og:type`/real dates, sitemap `lastmod`, nav tweaks).

The single highest-value finding: the installed `spatie/schema-org` v4.0.2 does **not** support array `@type` via the fluent `Schema::localBusiness()` builder (`getType()` returns one class short-name). The supported mechanism is the **`MultiTypedEntity` class** (verified in vendor source) — `new MultiTypedEntity()` then `->localBusiness(...)->homeAndConstructionBusiness(...)`, which merges all properties and emits `@type: ["LocalBusiness","HomeAndConstructionBusiness"]`. This is the exact pattern D-01 requires and the rewrite target for `LocalBusinessSchema::toScript()`.

Second key finding: vitrine images are referenced as static `asset('assets/brand/photos/*.jpg')` in Blade, **not** Vite-imported. So Vite imagemin/imagetools plugins (which rewrite import URLs) don't help here. The lightest correct approach is a **one-time/idempotent Artisan command using `spatie/image`** (already installed transitively, GD has `imagewebp`+`imageavif`, `cwebp`/`avifenc` present locally) that writes `.webp`/`.avif` siblings, plus a `<x-picture>` Blade component emitting `<picture>` source sets. `medialibrary` is the wrong tool for static assets (needs an Eloquent model + DB row per image). The og:image must stay a stable **absolute .jpg URL** for social crawlers.

**Primary recommendation:** Rewrite `LocalBusinessSchema` around `MultiTypedEntity` (D-01..D-03), author all long-form French copy from harvested Zyro content with explicit placeholders for missing facts (D-08/D-09), gate every city page on a real local fact (D-12), and ship image optimization as an Artisan command + `<picture>` component. Flag GBP creation + review solicitation as a tracked **zero-code Pierre-action** that unblocks D-05/D-06.

### Pattern 1: Multi-type LocalBusiness via `MultiTypedEntity` (D-01)
**What:** Emit `@type: ["LocalBusiness","HomeAndConstructionBusiness"]` with merged properties.
**When to use:** Replaces `Schema::plumber()` in `LocalBusinessSchema::toScript()`.
**Why this and not fluent builder:** `spatie/schema-org` v4.0.2 `BaseType::getType()` returns a single class short-name; the fluent `Schema::localBusiness()` can only emit a single `@type` string. `MultiTypedEntity::toArray()` collects each node's `@type` into `array_unique($types)` and merges all properties. [VERIFIED: vendor/spatie/schema-org/src/MultiTypedEntity.php lines 111–131]

```php
// Source: vendor/spatie/schema-org/README.md (multi-typed entities) + verified src
use Spatie\SchemaOrg\MultiTypedEntity;
use Spatie\SchemaOrg\Schema;

$mte = new MultiTypedEntity();

// Properties set on EITHER node are merged into one flat object with a combined @type array.
$mte->localBusiness(function (\Spatie\SchemaOrg\LocalBusiness $biz) {
    $biz->name('Dlo Azur Piscines')
        ->image(asset('assets/brand/photos/hero-pierre-piscine.jpg'))
        ->url(url('/'))
        ->telephone('+596696940054')
        ->email('contact@dloazurpiscines.com')          // D-03
        ->priceRange('€€')
        ->founder(Schema::person()->name('Pierre ADAM')) // D-02
        ->sameAs([])                                     // → GBP URL once live (D-09)
        ->address(
            Schema::postalAddress()
                ->addressCountry('FR')
                ->addressRegion('Martinique')
                ->addressLocality('Fort-de-France')      // locality only — NO street/postal (D-04)
        )
        ->geo(Schema::geoCoordinates()->latitude(14.6037)->longitude(-61.0594))
        ->areaServed([
            Schema::city()->name('Fort-de-France'),
            Schema::city()->name('Le Lamentin'),
            Schema::city()->name('Schoelcher'),
            Schema::city()->name('Les Trois-Îlets'),
            Schema::administrativeArea()->name('Martinique'),
        ])
        ->openingHoursSpecification([/* unchanged from current builder */]);
})->homeAndConstructionBusiness(); // second type, no extra props needed

return $mte->toScript(); // <script type="application/ld+json">{... "@type":["LocalBusiness","HomeAndConstructionBusiness"] ...}</script>
```
**Gotcha:** `MultiTypedEntity::add()` throws `TypeAlreadyInMultiTypedEntity` if you add the same class twice; use the chained method form. `toArray()` strips per-node `@context` and emits a single top-level `@context`. [VERIFIED: src lines 59–66, 133–145]

**Second wave (D-06):** add `->hasOfferCatalog(Schema::offerCatalog()->itemListElement([...Schema::offer()...]))` inside the `localBusiness` closure once the service list is stable.

## Common Pitfalls

### Pitfall 1: Assuming `Schema::localBusiness()->setType([...])` works for multi-type
**What goes wrong:** Devs try to pass an array to a single-type builder; output is `"@type":"LocalBusiness"` only, or an error.
**Why:** `getType()` reflects the class short-name (one string). [VERIFIED: BaseType.php:25]
**How to avoid:** Use `new MultiTypedEntity()`. Verify output with Google Rich Results Test after deploy.
**Warning signs:** `@type` renders as a string not an array in the rendered `<script>`.

### Pitfall 2: City pages tripping doorway/scaled-content enforcement
**What goes wrong:** 4 near-identical "[service] à [ville]" pages with only the city name swapped → spam classification, can drag the whole site.
**Why:** Post-March-2024 scaled-content-abuse + long-standing doorway policy; enforcement tightened through the Aug 26 2025 spam update. [CITED: developers.google.com/search/blog/2024/03/core-update-spam-policies]
**How to avoid:** D-12 hard gate — no city page publishes without ≥1 genuine operator-supplied local fact (water hardness, a real local job, commune-specific access constraint). Keep to exactly 4. City pages link → service pages (not reverse).
**Warning signs:** Two city pages differ only by the city token.

### Pitfall 3: Marking up an aggregateRating to "look established"
**What goes wrong:** Self-controlled rating → no rich result anyway (suppressed since 2019), and fabricated numbers risk a manual action stripping ALL rich results.
**Why:** Google does not show self-serving review snippets for LocalBusiness/Organization. [CITED: developers.google.com/search/blog/2019/09/making-review-rich-results-more-helpful; brightlocal.com/learn/review-schema]
**How to avoid:** D-05 — wait for real GBP reviews; surface them via the GBP entity, not page markup.

### Pitfall 4: Vite image plugin doesn't touch Blade `asset()` images
**What goes wrong:** Install a Vite imagemin plugin expecting `public/assets/*.jpg` to be optimized; nothing happens.
**Why:** Vite only processes assets that are **imported** in JS/CSS. Blade `asset()` references are opaque strings to Vite. [CITED: npmjs.com/package/vite-plugin-image-optimizer]
**How to avoid:** Use the Artisan/`spatie/image` path + `<picture>` component (recommended), or convert each image to a Vite import (high churn — reject).

### Pitfall 5: `llms.txt` served with wrong content-type / not at root
**What goes wrong:** File placed in a subdir or served as HTML.
**How to avoid:** `public/llms.txt`, plain markdown, served as `text/plain` or `text/markdown`. Mirror the existing `robots.txt` kernel route (`routes/web.php`) if a test assertion is desired. [CITED: llmstxt-spec — root path `/llms.txt`]
