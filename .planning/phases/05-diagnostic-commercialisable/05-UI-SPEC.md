## Spacing Scale

Exact Tailwind v4 scale (`@theme` + framework defaults). All multiples of 4 except the deliberate touch-target tokens. Use these tokens, never arbitrary `px`.

| Token | Value | Usage |
|-------|-------|-------|
| `1` | 4px | Icon gaps, inline micro-padding |
| `2` | 8px | Compact element spacing, chip padding-y |
| `3` | 12px | Stepper gap, tight label→field |
| `4` | 16px | **Default** element spacing, card inner gap |
| `6` | 24px | Card padding, section inner padding |
| `8` | 32px | Layout gaps, between result cards |
| `12` | 48px | Major section breaks within a step |
| `16` | 64px | Page-level vertical rhythm (hero, section bands) |

**Exceptions (deliberate, touch-target driven — DESIGN.md "cible tactile > 44px partout"):**

| Token | Value | Why |
|-------|-------|-----|
| `--spacing-13` (`h-13`) | 52px | Primary button + symptom-option height (one-handed, full sun) |
| `--spacing-15` (`h-15`) | 60px | Large hero CTA + large symptom tiles |
| `--spacing-18` (`h-18`) | 72px | Reserved (oversized field row on the poolside wizard) |
| stepper buttons | 48×56px (`h-12 w-14`) | Signature measure stepper, reachable at thumb |

Container: `--container-content` = `75rem` (1200px). Diagnostic working column caps at **`max-w-2xl` (~672px)** on the wizard/tree (single-focus, mobile-first); the landing hero spans the full content container.

Border radius (brand-rounded voice): `rounded-xl` 0.875rem (buttons, inputs), `rounded-2xl` 1.25rem (cards), `rounded-3xl` 1.75rem (feature/hero cards, the result "plan" card).

## Typography

Two families only. **Fredoka for words, Inter for numbers** (DESIGN.md named rule). ≥1.25× scale contrast between successive levels. Body capped at 65–75ch. **The 5 roles below are copied verbatim from `DESIGN.md` `typography:` / `.impeccable/design.json` / `resources/css/app.css @theme` — the project's committed design system.**

| Role | Family | Size | Weight | Line Height | Letter-spacing | Used for |
|------|--------|------|--------|-------------|----------------|----------|
| Display | Fredoka | `clamp(2.6rem, 5vw, 4rem)` | 700 | 1.05 | -0.005em | Hero title only — one per page. e.g. « Ta piscine te pose problème ? » |
| Headline | Fredoka | `clamp(1.875rem, 3vw, 2.5rem)` | 700 | 1.1 | -0.005em | Step section titles, the **diagnostic statement** on the result screen |
| Title | Fredoka | `1.25rem` | 600 | 1.3 | — | Result-card titles (Problème), symptom-tile labels, lead-form heading, **inline field labels / inline emphasis** (use Title weight, not a new size) |
| Body | Inter | `1rem` | 400 | 1.6 | — | All prose, disclaimer body, plan steps, **field hints (« ex: 7,4 ») and captions** (Body, or the Label role below — no separate 0.875rem size) |
| Label | Inter | `0.75rem` | 700 | 1.2 | 0.18em | uppercase kicker in `lagon-600` above titles; confidence-index label; status chips; **secondary-action / caption micro-text** |

**Numeric is a rendering variant, not a 6th size.** Apply `font-variant-numeric: tabular-nums` to whichever role renders the value (Body / Title / Display): every pH/chlore/TAC/sel value, every dose, totals. The stepper central field renders the value at Fredoka 700, `1.5rem`, `tabular-nums` per the signature `stepper` primitive in `design.json` — this is the rendered instance of that component, not a new type token.

### Type-scale fidelity & override (deliberate)

- The **5 roles (display / headline / title / body / label) and 3 weights (400 / 600 / 700)** above are copied verbatim from `DESIGN.md` (`typography:` block), `.impeccable/design.json` (`typographyMeta` + primitives), and `resources/css/app.css` (`@theme`) — the project's **established, committed design system**, authored via the `impeccable` skill, which the commanditaire **mandated** using for this phase.
- The gsd-ui-checker's generic heuristic caps (≤4 sizes, ≤2 weights) are **explicitly and deliberately overridden here.** Reducing below the project's real system would create **design drift across the whole app** (vitrine, app pro, portail client all share these tokens) and violate the directing mandate. `impeccable`'s own typography law calls for hierarchy via **scale + weight contrast at ≥1.25× steps**, with no 4-size / 2-weight cap.
- **Display vs Headline are both required and distinct:** Display is the hero tier (one per page, `clamp(...4rem)`); Headline is the in-flow step / section + diagnostic-statement tier (`clamp(...2.5rem)`). **Title / Body / Label each carry a distinct job** (card / section titles + inline emphasis · prose + hints · uppercase kicker + status / caption). None is decorative.

2. Active/selected state of a symptom tile or a decision-tree option (ring + tint, not fill-everything).
3. The current step indicator in the wizard progress affordance.
4. Inline links and the focus ring (`focus-visible` 2px azure, already global in app.css).

Everything else uses neutrals (sand/ink) and the **semantic palette below**. The lead-capture submit is azure; the WhatsApp hand-off is **WhatsApp green** (channel color, intouchable) — these two never compete because they live on different surfaces (form vs escalation peak).

### Confidence index (CDC §5.5 — a free "intelligence" signal)

Three levels, chip + one-line explanation:
- **« Confiance élevée »** (`success`) — « basé sur tes mesures. »
- **« Confiance moyenne »** (`warn`) — « affine en mesurant le stabilisant / le TAC. »
- **« Indicatif »** (`warn`) — « diagnostic visuel sans mesure — pour confirmer, mesure ton eau ou demande à Pierre. »

### « Qu'as-tu déjà essayé ? » (the differentiator — BLUEPRINT §6)

Multi-select chips, label: **« Qu'as-tu déjà tenté ? (pour ne pas te reproposer ce qui n'a pas marché) »**. Options e.g. « Chlore choc », « Brossage », « Anti-algues », « J'ai ajusté le pH », « Rien encore ». A tried-and-failed action is pruned from the plan and used as a diagnostic signal.

## Surface-by-Surface Interaction Contract

Each surface lists: register · register-justified color · key states · interaction · motion · the components it reuses/creates. Mockup intent is re-expressed in the project design system — **do not copy the mockup's React code** (it is a minified Next.js export; flows/copy/formulas were already extracted verbatim into `05-RESEARCH.md`).

### S1 — Public `/diagnostic` landing + entry  (register: **brand**)
- **Color:** Drenched. Marine `navy-900`/`950` hero ground *or* an azure full-bleed band with a real Martinique pool photo (`.photo-grade`, overlay marine for text contrast). Sun accent allowed on the « Ma piscine est verte ? » warm tile only.
- **Layout:** Full content container. Display title (one) + one-line promise + **two large entry tiles** (≥`h-15`): « Trouver mon problème » (symptom) and « Analyser mon eau » (chemistry). Mobile-first single column; tiles stack. Indexable SEO page (extends 999.1).
- **States:** default; (returning visitor) a quiet « Reprendre mon dernier diagnostic » strip if the local carnet has entries.
- **Interaction:** tile tap → Alpine sets `mode` + advances to the disclaimer-aware first step (no full navigation). The hero CTA in vitrine nav routes here.
- **Motion:** `.rise` entrance (transform+opacity, 700ms `ease-out-quint`), staggered tiles `calc(var(--i)*50ms)` capped. No layout-property animation. Respect `prefers-reduced-motion` (crossfade).
- **Reuse / create:** reuse `x-button` (primary/whatsapp), `x-icon.*`, `x-cta-whatsapp`, vitrine nav partial, `x-seo.meta`. Create: `diagnostic` hero section in `resources/views/vitrine/diagnostic.blade.php`.

### S2 — Symptom decision-tree flow  (register: **product**)
- **Color:** Restrained. Sand surfaces; selected option = azure ring + `azure-50` tint. Emoji from the mockup tree (🟢⚪🟤💎⚡) as quick visual anchors on entry nodes.
- **Layout:** Single question per screen, capped `max-w-2xl`. Big option buttons (`h-13`+), one decision per view (progressive disclosure — symptôme d'abord). A subtitle line under questions that need it (e.g. brown-2 metals hint).
- **States:** question node; the **« Qu'as-tu déjà tenté ? »** multi-select node (chips); branching; result leaf (→ S5). Back affordance always present.
- **Interaction:** Alpine traverses the tree (config PHP array surfaced to the view; **no dose formula in JS**). Selecting an option advances client-side. The `auto-green` node has the one conditional branch (chlore OK + pH>7.6 → ph-calc, else algues-installées) — handled by Alpine over server-provided node data.
- **Motion:** horizontal slide between nodes (transform), 200–300ms; back = reverse, ~75% duration.
- **Reuse / create:** create option-tile + chip-multiselect partials (use existing chip styling `ds-chip-azure` / status chips). Reuse `x-icon.arrow-right`, `x-icon.check`.

### S3 — Water-chemistry wizard  (register: **product**)
- **Color:** Restrained. The **signature stepper** ( `[−][ value ][+]` ) carries azure on `+` only.
- **Layout (2 steps, per SPEC Req 2):**
  - **Step 1 — pool info:** size mode toggle (Volume m³ *or* surface + profondeur → derived volume shown live), **filter-type select** (sable/verre · cartouche · diatomées — *constrained select*, mandatory before any product reco per FLOCULANT-BRANCH-SPEC §2; pre-fill from `piscine.filtration` for logged-in clients with override, treat free-text pool value as non-authoritative hint per CONTEXT D-08 gap), « Piscine au sel ? » (oui/non + ppm).
  - **Step 2 — measurements:** pH, chlore libre, alcalinité (TAC), stabilisant, taux de sel — each as a stepper OR numeric field with inline helper (« ex: 7,4 ») and `is_numeric` validation. `tabular-nums`. Optional (V0-light, audit P1 intake): chlore total, TH.
- **States:** default; per-field error (« La valeur doit être un nombre. Exemple : 7,4 »); derived-volume live readout; step indicator (1 / 2).
- **Interaction:** **Alpine drives step nav — no network round-trip per step (D-01)**. Only « Calculer mon plan d'action » is a `wire:click` server call (DoseEngine). Keep step index purely in Alpine (RESEARCH Pitfall 1 — never bind to Livewire). `inputmode="decimal"` on numeric fields; accept comma decimal (« 7,4 »).
- **Motion:** field focus ring 200ms; stepper press `:active scale(.95)` (the one sanctioned scale, on the stepper button only, per design.json). Compute submit shows a reassuring loading state (« Calcul de ton plan… »), preemptive (impeccable perceived-performance).
- **Reuse / create:** reuse the **stepper** primitive (design.json `stepper-plus/minus`), `input` primitive, error pattern from ContactForm. Create: filter-type select, size-mode toggle, the wizard step shell.

### S4 — Disclaimer acceptance gate  (register: **product**)
- **Color:** Calm. Card on sand, azure primary accept button. Not alarmist.
- **Affordance decision (impeccable "no modal-as-first-thought"):** the disclaimer is an **inline progressive step**, shown **before any dosing/chemical output** (timing per CDC §4 — at the boundary to the first numeric/chemical result, NOT at page entry, so a purely visual symptom diagnosis stays friction-free). Justification for not using a modal: a modal interrupts and reads as a cookie-banner reflex; an inline step keeps the user in the linear flow and makes acceptance a deliberate, legible action (DIAG-03 requires *explicit* accept). A modal is only acceptable if a flow can reach dosing from multiple branches and an inline step can't cover them — not the case here (single Livewire flow).
- **States:** unaccepted (continue disabled / hidden); accepted (`disclaimer_accepted_at` set server-side on persist, D-04). Server guard re-checks (RESEARCH Pattern 4) — never trust the client gate alone.
- **Interaction:** explicit checkbox/affirm + « J'ai compris, voir les recommandations ». Advancing to any dose/chemical screen without it is impossible (Alpine hides + Livewire rejects).
- **Reuse / create:** reuse callout/card + checkbox styling; create disclaimer step partial. Disclaimer copy library reusable across the safety, baignade-delay, dose-indicative, escalation, legal-accept variants (CDC §7).

### S5 — Results  (register: **product**)
- **Color:** Restrained base; **meaning-colored** chips (confidence) + ambre safety block + danger hard-stop. Plan card = `rounded-3xl` feature card.
- **Layout (gabarit fixe, CDC §7 — same order every time):** ① Diagnostic statement (Fredoka headline) + **confidence chip** → ② Pourquoi (one pedagogical line) → ③ **Safety block** (ambre, before any chemical step) → ④ Étapes ordonnées (safe gestures first) → ⑤ **Dosage** cards (only if measurements) with « Re-teste avant la dose suivante » → ⑥ Quand appeler Pierre (escalation seam). Result cards render as **Problème / Étapes / Dosage / Produit** (CONTEXT specifics).
- **Action-aware (the differentiator, BLUEPRINT §6 / audit):** a gesture the user marked as already-tried-and-failed is **never re-proposed**; failure routes to chlore-lock / metals / calcaire / resistant-algae messaging. This is rules in PHP (0 infra) — the "intelligence" that justifies the whaou without backend cost.
- **States:** dosed plan; no-adjustment-needed (« Ton eau est équilibrée », success); hors-DIY preemptive (collapses ④/⑤, surfaces escalation — see S6); ambiguous/insufficient → invite to measure or escalate.
- **Cards rule (impeccable "no identical card grid"):** the plan is an **ordered vertical list of differentiated blocks**, not a uniform N-column card grid. Dosage cards differ visually from step rows (numeric-led, `tabular-nums`).
- **Motion:** sequential reveal of plan blocks (stagger, capped); confidence chip a quiet fade-in. No bounce.
- **Reuse / create:** reuse status chips, offline-banner styling (for the ambre safety block), `x-icon.shield`/`check`/`sparkle`. Create: result-plan card, dosage card, confidence chip variants.

### S6 — One-gesture rich escalation → WhatsApp  (register: **brand** — the conversion peak)
- **Color:** Committed/warm. A marine or azure card framing the official **WhatsApp green** CTA. This is the emotional peak (BLUEPRINT §5) — it earns more visual weight than a normal product surface.
- **Layout:** A context recap (what Pierre will receive) + the single CTA. No long form.
